# 核心意图（一定要保持不偏离）

这套方法的终极目标是：**让学习过程中产生的“结构”稳定下来**，并且能长期迭代。

这里的“稳定”指两件事：

1) 你每次复习都能沿着同一条路径回忆：从 UI 的空间布局出发（左→右，上→下）。  
2) 你新增/调整内容不会让整体结构坍塌：上层索引始终轻量，下层细节可无限扩展。

所以在任何自动化/半自动化的操作里，都要优先保证：

- **白板（Canvas）只承担“视觉索引 + 顺序记忆”**，不要把大量解释塞进白板。
- **Markdown 只承担“层级拆分 + 细节沉淀”**，上层页只做导航，下层页再写细节。
- **顺序必须稳定**：白板排布与索引组织，始终遵循“从左到右、从上到下”。

如果生成结果看起来“更漂亮”但破坏了顺序/层级，那就是失败。

# 方法框架（做什么、不做什么）

## 1) 白板（Canvas）应该包含什么

白板节点只放：

- 软件界面截图（最大节点）
- 功能区/模块截图（菜单栏、快捷工具栏、模块窗口、视图工具栏……）
- 从截图引出的**索引页链接**（指向 Markdown 的上层页）

重要提醒：截图需要手动完成（先截图，再放进 Vault，再在白板里引用/排布）。自动化主要帮助你做“结构骨架与连线”，不替代截图采集本身。

白板尽量不放：

- 大段文字解释（文字一多就变成“墙”，复习效率反而下降）
- 对单个命令的详细说明（应该下沉到 Markdown）

## 2) Markdown（笔记）应该如何分层

推荐层级（从上到下）：

- 软件根（例如 `SolidWorks.md`）
- 模块索引（例如 `模块窗口.md`）
- 选项卡/功能区索引（例如 `草图.md / 特征.md / 曲面.md`）
- 分区索引（当命令很多时才需要；例如把几十个命令先归成 3~7 个区）
- 叶子页（单命令/单主题：参数、步骤、注意点、坑、例子、技巧）

关键约束：

- **上层页 = 目录/导航页**：只放双链清单与必要的“相关链接”，不要写长段细节。
- **下层页 = 细节页**：具体写怎么用、怎么选、怎么排错、怎么复现、怎么总结。

## 3) “命令很多”时的处理：先分区，再拆页

当一个页面要列出很多命令时，不要硬塞一个长清单。

正确做法：

1) 先根据 UI 分组或语义聚类，建立 3~7 个“分区”（名字要稳定、好记、能长期使用）
2) 每个分区做一个索引页（只列双链）
3) 分区下面再拆命令叶子页

这样你的复习路径是稳定的：先定位“属于哪一组”，再进入组内。

# 命名与排序规则（非常重要）

为了让结构可机器生成、可长期维护，命名与排序必须可预测。

推荐命名：

- 采用 `NN-中文名.md` 的格式（`NN` 是两位数字，保证排序稳定）
- 叶子命令页如果很多，采用 `NN-MM-中文命令.md`

推荐排序：

- 目录页内的双链清单按 UI 的“左→右，上→下”顺序列
- 白板节点也按同样顺序摆放（不是随机摆放）

# 图片管理规则（避免“数字文件名地狱”）

目标：让截图资产**可读、可移动、可复用**，并且不会因为 Obsidian 自动命名导致后续难维护。

- 图片统一收纳到软件目录内的一个固定位置，例如：`02-知识图谱/SolidWorks/99-图片/`
- 白板里某个“功能区/选项卡”如果只对应**一张**截图，优先把截图重命名为该项名称（例如 `特征.png / 草图.png / 曲面.png`）
- 移动/重命名后必须同步更新 `.canvas` 里的图片路径（否则白板会显示找不到）

## 操作原则（避免意图偏差）

当 Codex/自动化要“帮你生成结构”时，必须遵守：

- 不要把你整个 Obsidian 仓库推上 GitHub（只导出最小子集）
- 不要替你“重新设计分类学”（先复用你已经形成的稳定分区与命名）
- 不要为了省字而把“关键约束”删掉（尤其是顺序规则、上层页/下层页职责分离）

# 通用迁移：换软件怎么做

迁移到其它软件时，只替换“截图来源”和“模块名称”，规则保持不变：

1) 先用截图搭 UI 地图（白板）
2) 再用 Markdown 拆命令层级（目录页轻、叶子页细）
3) 固定顺序（左→右，上→下）
4) 命令多就先分区再下钻
5) 技巧/反思按时间戳沉淀，并挂回对应功能区
