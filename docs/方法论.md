# 方法论：用 Obsidian 学复杂软件（白板索引 + 分层笔记）

这份方法论的目标不是“把命令抄一遍”，而是让你在学习过程中持续构建一个**可记忆、可检索、可扩展**的软件知识结构。

## 1. 结构总体：两条线并行

### 1.1 白板（Canvas）：只负责“视觉索引 + 顺序”

白板节点只放两类东西：

1) **截图**（软件界面、某个功能区、某个面板）  
2) **指向笔记的链接**（从截图引出对应的 Markdown 索引页）

白板不承担“写解释/写大量文字”的职责。  
它的核心价值是：你每次打开白板，都能马上回到软件界面那种直觉式结构里。

### 1.2 Markdown：只负责“层级拆分 + 细节沉淀”

Markdown 里做两件事：

- 建立**层级目录**（上层页）  
- 对单个命令/单个主题写**可复用细节**（下层页）

上层页的内容保持克制：只做“目录 + 导航 + 相关链接”。  
下层页再写具体：参数、使用步骤、注意点、坑、例子、自己的反思总结。

## 2. 分层规则：从大到小，但不要“文件夹地狱”

你之前的“文件夹一级一级往下建”会遇到两个问题：

- 结构不够直观：打开文件夹看不出 UI 的空间结构。
- 图谱随机：关系图谱节点位置不稳定，记忆路径被破坏。

现在的做法是：把“结构感”交给白板，把“层级感”交给 Markdown。

**推荐分层：**

- 软件（根）：`SolidWorks.md`
- 模块（功能区）：如 `模块窗口.md`
- 选项卡/页面：如 `草图.md / 特征.md / 曲面.md`
- 分区（可选）：当命令很多时，为了减轻认知负担，先分成几个区
- 命令（叶子）：一个命令一页（或一类非常相近的命令一页）

## 3. 白板排版规则：顺序必须稳定

白板节点排布的目的不是“好看”，而是提供稳定的记忆路径。

**固定顺序：从左到右、从上到下。**

- 你看到的 UI 也是这样读的；
- 你回忆的时候也会沿这条路径走；
- 你复习时也会更快定位到“自己上次停在哪里”。

不要依赖“随机布局 + 关系线”来记忆（那会造成漂移）。

## 4. 何时用截图，何时用文本

### 4.0 截图怎么来（必须提醒）

这套方法里，“截图”是结构化学习的入口，但**截图目前需要手动完成**（在软件里截 UI / 功能区 / 面板，再放进 Vault）。

因此在执行上要有一个心理预期：

- 搭“UI 地图”（白板）阶段会有一段明显的手动工作量；
- 一旦地图搭好，后续学习会越来越偏向“写 Markdown 细节 + 迭代技巧”，效率会明显提升。

### 4.1 用截图

当信息**视觉上更直观**，且文字描述会变得冗长时：

- 工具栏区域
- 图标密集且名称相似的命令组
- 某个面板/窗口布局

### 4.2 用文本

当你开始进入“可操作的步骤/参数/注意事项”时：

- 单个命令的用法
- 参数含义与常见组合
- 使用顺序、前置条件
- 易错点与排错
- 你自己的建模技巧、反思与总结

## 5. 命令很多怎么办：先分区，再拆页

当一个功能区有太多命令，直接列清单会让你在复习时感到“密密麻麻无从下手”。

做法是：

1) 先把大量命令按 UI 或语义分成若干区（分区名要稳定、好记）
2) 每个区单独一页做目录
3) 具体命令再拆成独立页面

这样你复习时会先回忆“这块属于哪一组”，再进入组内定位。

## 6. 知识如何迭代：技巧用时间戳，挂回索引

学习过程中会出现大量“技巧/心得/踩坑”。

建议不要把它们都塞进一个大杂烩文档里，也不要把白板塞满文字。

推荐：

- 每条技巧一个小条目（可以带时间戳）
- 放在对应功能区的“建模技巧”里
- 从白板的功能区截图引出技巧索引页（或从功能区索引页链接到技巧页）

你需要的是：**技巧永远能回到它发生的上下文里**。

## 7. 通用迁移：换软件也一样

换成任何其它软件（CAD/CAE/渲染/剪辑/IDE 等），规则不变：

- 先用截图把 UI 地图搭起来（白板）
- 再用 Markdown 把命令体系拆清楚（层级）
- 通过稳定顺序与双链来保证复习效率

SolidWorks 只是一个例子；真正开源的是这套“学习与记忆的软件结构化方法”。
